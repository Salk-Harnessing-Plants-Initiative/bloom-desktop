# Design: Update Scan Directory Path Format

## Context

Bloom-desktop must produce scan directories compatible with the pilot application and any downstream tools. The pilot stores scans as `YYYY-MM-DD/<plant_id>/<uuid>/` relative to the configured `scans_dir`. Bloom-desktop currently uses `<experiment_id>/<plant_qr_code>_<timestamp_ms>/`.

## Goals

- Match pilot directory format for data portability
- Store relative paths in `Scan.path` and `Image.path`, matching pilot
- Preserve bloom-desktop's path sanitization for security

## Non-Goals

- Migrating existing scans (none in production)
- Changing bloom-fs upload logic (already uses flat paths)
- Moving path construction from renderer to main process (defer to future refactor)

## Decisions

### 1. Directory Format: Pilot format

| Aspect | Pilot (`scanner.ts`) | Bloom (current) | **Chosen** |
|---|---|---|---|
| Structure | `YYYY-MM-DD/<plant_id>/<uuid>` | `<experiment_id>/<plant_qr_code>_<ts>` | `YYYY-MM-DD/<plant_qr_code>/<uuid>` |
| First level | Date (human-readable) | Experiment UUID (opaque) | Date |
| Uniqueness | UUID (guaranteed) | Timestamp ms (near-unique) | UUID |

**Rationale**: Date-first is human-friendly for browsing scans on disk. The experiment UUID is opaque — it belongs in metadata.json (#99), not the directory name. UUID guarantees uniqueness.

### 2. UUID Source: Separate `uuidv4()`, matching pilot

The pilot generates a **new** `uuidv4()` for each scan (`scanner.ts:47`) and uses it as both the directory name and the scan's metadata ID. The database `Scan.id` is a separate UUID generated by Prisma's `@default(uuid())`.

We follow the same pattern: generate `uuidv4()` in `CaptureScan.tsx` before `scanner.initialize()`, pass it in metadata so `scanner-process.ts` can use it.

### 3. Path Storage: Relative paths for both Scan.path and Image.path

**Pilot behavior** (confirmed from code):
- `Scan.path` = `"2026-03-04/PLANT-001/abc-uuid"` (relative, `scanner.ts:211`)
- `Image.path` = `"2026-03-04/PLANT-001/abc-uuid/001.png"` (relative, `scanner.ts:89`)

**Current bloom-desktop behavior**:
- `Scan.path` = `"/Users/x/.bloom/scans/exp-uuid/PLANT-001_1709500000000"` (absolute, `scanner-process.ts:235`)
- `Image.path` = `"/Users/x/.bloom/scans/exp-uuid/PLANT-001_1709500000000/001.png"` (absolute, `scanner-process.ts:202`)

**Change**: Store relative paths. Code that accesses files must prepend `scans_dir`:
- `scanner-process.ts:235` — derive relative path by stripping `scans_dir` prefix from `scanResult.output_path`
- `scanner-process.ts:202` — build `Image.path` as relative: `path.join(relativeScanPath, file)`
- `ScanPreview.tsx:342` — `pathToFileUrl(scansDir + '/' + currentImage.path)` (needs scans_dir from config)
- `image-uploader.ts:243` — prepend scans_dir to image paths before passing to bloom-fs

### 4. Date Formatting: Local timezone YYYY-MM-DD

Port the pilot's `getLocalDateInYYYYMMDD()` (`scanner.ts:321-327`):
```typescript
function getLocalDateYYYYMMDD(date: Date = new Date()): string {
  const timeZoneOffsetInMs = date.getTimezoneOffset() * 60000;
  const localDate = new Date(date.getTime() - timeZoneOffsetInMs);
  return localDate.toISOString().slice(0, 10);
}
```

### 5. Sanitization: Apply to plant_qr_code only

The date and UUID segments are code-generated (safe). Only `plant_qr_code` comes from user input and needs `sanitizePathComponent()`.

### 6. How scans_dir reaches consumers

- **Renderer** (CaptureScan, ScanPreview): Already loads `scans_dir` from `window.electron.config.get()` (see `CaptureScan.tsx:138-146`)
- **Main process** (scanner-process, image-uploader): `scans_dir` comes from `ScannerSettings.output_path`. For image-uploader, it will need access to scans_dir from config.

## Risks / Trade-offs

- **Relative paths require scans_dir context**: Every consumer that accesses files must have access to `scans_dir`. This is already available in both renderer (config) and main process (scanner settings/config).
- **Image-uploader needs config access**: Currently reads absolute `Image.path` directly. Will need to load `scans_dir` from config to construct absolute paths. This is a small but necessary change.

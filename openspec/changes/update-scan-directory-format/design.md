# Design: Update Scan Directory Path Format

## Context

Bloom-desktop must produce scan directories compatible with the pilot application and any downstream tools. The pilot stores scans as `YYYY-MM-DD/<plant_id>/<uuid>/` relative to the configured `scans_dir`. Bloom-desktop currently uses `<experiment_id>/<plant_qr_code>_<timestamp_ms>/`.

## Goals

- Match pilot directory format for data portability
- Preserve bloom-desktop's path sanitization for security
- Store portable (relative) paths in the database

## Non-Goals

- Migrating existing scans (none in production)
- Changing bloom-fs upload logic (it already uses flat paths)
- Moving path construction from renderer to main process (defer to future refactor)

## Decisions

### 1. Directory Format: Pilot format

| Aspect | Pilot | Bloom (current) | **Chosen** |
|---|---|---|---|
| Structure | `YYYY-MM-DD/<plant_id>/<uuid>` | `<experiment_id>/<plant_qr_code>_<ts>` | `YYYY-MM-DD/<plant_qr_code>/<uuid>` |
| First level | Date (human-readable) | Experiment UUID (opaque) | Date |
| Uniqueness | UUID (guaranteed) | Timestamp ms (near-unique) | UUID |
| Browsability | Easy chronological browsing | Grouped by experiment | Chronological |

**Rationale**: Date-first is more human-friendly for browsing scans on disk. The experiment UUID is opaque to humans and provides no browsing benefit as a directory name — it belongs in metadata.json instead. UUID guarantees uniqueness without risk of millisecond collisions.

### 2. Path Storage: Relative paths

Store `YYYY-MM-DD/<plant_qr_code>/<uuid>` in `Scan.path` (relative to `scans_dir`), matching pilot behavior. This makes paths portable across machines and environments.

Currently, bloom-desktop passes the full absolute `outputPath` to `scanner.initialize()`. The path stored in the database should be the relative portion only.

### 3. Date Formatting: Local timezone YYYY-MM-DD

Use the pilot's approach: offset `Date` by timezone to get local date, then format as `YYYY-MM-DD`. This ensures scans taken on the same calendar day (in the user's timezone) are grouped together.

### 4. Sanitization: Apply to plant_qr_code only

The date segment is generated by code (safe). The UUID is generated by code (safe). Only the `plant_qr_code` comes from user input and needs sanitization via `sanitizePathComponent()`.

### 5. Scan UUID Generation

Generate the scan UUID in the renderer (via `crypto.randomUUID()` or `uuidv4()`) before calling `scanner.initialize()`. This UUID serves as both the directory name and is passed as part of the scan metadata. The database `Scan.id` (already a UUID) could potentially be reused, but generating a separate one keeps concerns separated.

## Risks / Trade-offs

- **Date grouping vs experiment grouping**: Users lose the ability to browse by experiment on disk. Mitigated by: experiment info in metadata.json, and the Browse Scans UI provides filtering by experiment.
- **Relative vs absolute path**: Code that reads `Scan.path` must prepend `scans_dir`. This matches pilot behavior and is more portable.

## Open Questions

- Should we reuse `Scan.id` (the database UUID) as the directory UUID, or generate a separate one? Recommendation: reuse `Scan.id` for simplicity, but this depends on whether the ID is available before the scan directory is created.
